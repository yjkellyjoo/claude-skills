# commit

Turn the current working tree into a series of small, clean, feature-by-feature commits. Conventional Commits subject, comprehensive bullet body, signed off and co-authored. Never one big dump.

Invoking this skill **is** the request to commit — proceed without asking "should I commit?". The only confirmations that remain are: a suspected secret/artifact (flag it), and a hook failure (stop and report).

## 1. Inspect

Run, and read enough to understand the intent of every change:

```bash
git status --porcelain
git diff            # unstaged
git diff --staged   # staged
git log --oneline -20
```

- If the tree is **clean** → report "nothing to commit" and stop.
- Use `git log` to learn the repo's existing scope vocabulary (the `(scope)` part), so your commits match local convention.

## 2. Respect an existing stage

If something is **already staged** when invoked, treat it as explicit intent:

1. Author a proper conventional message for exactly what's staged and commit it **first**.
2. Then continue grouping the remaining unstaged + untracked changes (steps 4+).

If nothing is staged, do the full analyze-and-split.

## 3. Branch if on the default branch

Check the current branch (`git branch --show-current`).

- If on **`main`** or **`master`**: create and switch to a feature branch **before** the first commit, named `type/short-slug` (slug from the dominant change — e.g. `feat/commit-skill`, `fix/api-error-codes`). Announce the branch name.
- If already on a feature branch: **stay there**.
- One branch holds the whole commit sequence — do **not** branch per commit.

```bash
git switch -c feat/short-slug   # only when on main/master
```

## 4. Group feature-by-feature

Partition the changes into logical commits:

- Separate unrelated features; refactors vs. behavior changes; docs/config vs. code.
- Keep one coherent change together.
- **Untracked files are real work** — include them, but add by **explicit path** (never `git add -A` / `git add .`).
- Leave out and **flag** anything that looks like a secret or build artifact (`.env`, `*.key`, `node_modules/`, `dist/`, `build/`). Do not commit it; tell the user it was skipped.

## 5. Stage precisely

- File-level groups: `git add <path> <path>`.
- When a single file spans multiple commits: `git add -p <path>`, or write a patch and `git apply --cached patch.diff`.
- Interactive `git add -i` and `git rebase -i` are **unavailable** in this environment — drive everything via explicit pathspecs and patch files.

## 6. Message format (the core)

```
type(scope): subject

- bullet describing one change, comprehensive but concise
- another change in this commit
optional short summary sentence

Signed-off-by: Name <email>
Co-Authored-By: Claude <model> <noreply@anthropic.com>
```

**Subject** — Conventional Commits:
- Form `type(scope): subject`. Scope is optional but use it when there's a clear area.
- **Lowercase imperative**, no trailing period.
- Allowed types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`.

**Body** — blank line after subject, then `-` bullets, each a comprehensive-but-concise description of one change. An optional short summary line may follow the bullets.

**Trailers** (blank line above them):
- `Signed-off-by:` — generated by committing with `-s`; it reads `git config user.name` / `user.email`. Do **not** hardcode the identity.
- `Co-Authored-By: Claude <model> <noreply@anthropic.com>` — `<model>` is the model **currently running this skill** (e.g. `Opus 4.8`, `Sonnet 4.6`), not a fixed string. You know your own model from your environment context — fill it in each time so the trailer reflects whoever actually made the commit.

Commit with the body + Co-Authored-By passed as the message and `-s` adding the sign-off:

```bash
git commit -s -m "type(scope): subject" -m "- bullet one
- bullet two

Co-Authored-By: Claude <model> <noreply@anthropic.com>"
```

(`-s` appends `Signed-off-by:` from git config; keep `Co-Authored-By` at the end of the message body so both land as trailers.)

## 7. Commit the sequence straight through

Print each commit's plan as you make it (so it's legible in scrollback). **No per-commit confirmation gate** — execute the whole sequence. If a grouping is ever wrong it's trivially `git reset`-able. The only mid-sequence hard-stop is a suspected secret/artifact you left out (step 4) — flag it.

## 8. Hooks

- **Never** use `--no-verify`.
- If a pre-commit hook **auto-fixes / restages** files (e.g. a formatter), re-stage the modified paths and retry that one commit **once**.
- If a hook **fails outright**, stop the sequence, report which commit failed with the hook output **verbatim**, and leave already-made commits intact. Do not plow ahead or try to fix the lint/test error unless asked.

## 9. Finish

```bash
git log --oneline -n <count>
```

If you created a branch, note: ``on `type/x`; push when ready``.

**No push, no PR** — out of scope (a separate concern).

## Guardrails

- Commit only when asked — invoking this skill is the ask.
- Never commit secrets or credentials; flag and skip them.
- Report failures honestly with real output; never claim a commit succeeded if it didn't.
